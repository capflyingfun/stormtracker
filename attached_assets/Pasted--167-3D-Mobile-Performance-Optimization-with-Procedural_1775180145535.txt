#167 - 3D Mobile Performance Optimization with Procedural Clouds

What & Why

With 67-84 storm zones, the 3D view bogs down on mobile. Per-cell object counts are too high: each zone spawns 5-7 scene objects plus setInterval timers. The main GPU costs are PointLights (expensive per-fragment passes), per-frame rain particle animation, unique CanvasTexture per cell, and lightning timer thrashing. This task replaces the GLB cloud model with lightweight procedural clouds and applies targeted optimizations to keep performance smooth at 100+ zones on mobile.

Done looks like

- 3D view stays fluid at 60+ zones on mobile (no visible stutter on iPhone LTE)
- Clouds are procedurally generated, semi-transparent, and colored by tier
- Visual quality preserved — clouds glow via vertex color, rain still falls, lightning still flashes
- Glow slider still controls intensity
- No GLB cloud model loading

Out of scope

- Changing the tier tint color scale or sky/lighting systems
- Desktop-specific optimizations (focus on mobile)
- Changing radar data fetch or zone logic

Tasks

1. Replace GLB cloud with procedural merged geometry — For each cell, generate a cloud by merging 3-5 scaled/offset low-poly spheres (8x6 segments each, ~480 tris total) into a single BufferGeometry using BufferGeometryUtils.mergeGeometries(). This replaces the imported GLB model (~2K polys). Use vertex colors to bake the tier tint + dBZ intensity directly into the geometry — no PointLight needed for glow. Apply MeshBasicMaterial with vertexColors: true, transparent: true, opacity: 0.8, depthWrite: false. Set renderOrder so clouds draw after opaque terrain but before rain particles. Wire the glow slider to a color brightness multiplier on the vertex colors (lerp toward white at higher glow levels). Remove the GLB loader and all per-cell PointLight glow creation.
1. Consolidate lightning into a single timer — Replace all per-cell setInterval lightning timers with a single requestAnimationFrame-based lightning system. Each frame (or every 2nd frame), pick random eligible cells (>=40 dBZ) to flash. Cap concurrent flashes to 3 on mobile. Use emissive color flash on the cloud material or a brief vertex color brightness spike instead of creating/removing temporary PointLights. This eliminates 40-60 simultaneous setInterval timers and the GC pressure from rapid light creation/destruction.
1. Share halo and label textures — Create one shared halo CanvasTexture per tier (6 total) instead of one per cell. For labels, create shared textures for common dBZ value ranges. This cuts 60+ unique GPU texture uploads down to ~12.
1. Throttle rain animation and cap rain cell count — Only animate rain particles every 2nd frame instead of every frame. On mobile, cap rain rendering to the nearest 20 cells (skip distant ones). Reduce max rain particle count per cell from 120 to 60 on mobile.
1. Add distance-based LOD culling — Skip rendering rain, halos, labels, and lightning for cells beyond a configurable distance threshold (e.g. 40km on mobile). Only render the cloud mesh and cone outline for distant cells. Use Object3D.visible = false for culled objects rather than adding/removing from the scene graph (toggling visibility is cheaper than scene mutations). Throttle the distance sort to every ~500ms, not every frame.

Performance notes

- Do NOT switch to MeshStandardMaterial — it adds PBR lighting calculations that are more expensive on mobile. Stick with MeshBasicMaterial and simulate glow via brighter vertex colors.
- With transparent clouds (opacity 0.8, depthWrite false), internal PointLights would look wrong bleeding through faces. Baked vertex colors are the correct approach.
- The procedural cloud at ~480 tris vs GLB at ~2K tris saves roughly 100K polys across 67 cells.
- Target: under 100 draw calls for clouds, zero PointLights for glow, zero per-cell setInterval timers.
