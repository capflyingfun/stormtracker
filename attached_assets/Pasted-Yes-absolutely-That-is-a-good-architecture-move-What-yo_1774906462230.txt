Yes, absolutely. That is a good architecture move 👍🌩️

What you’re really describing is turning location handling into a single organized system instead of scattered little one-off behaviors. That usually makes apps like StormTracker much easier to debug and expand.

Better than just an array

You can use arrays, but for this job I’d recommend a registry/object of methods instead of plain arrays.

Why:
	•	an array like ["GPS", "Address", "Map", "Favorites"] only stores names
	•	but each method also needs:
	•	label
	•	how to fetch location
	•	how to save location
	•	whether it needs user input
	•	whether it should reverse geocode
	•	fallback behavior
	•	icon / UI text

So instead of just:

const Set_Location = ["GPS", "Address", "Map", "Favorites"];

do something more like:

const LOCATION_METHODS = {
  GPS: {
    key: "GPS",
    label: "Use GPS",
    needsInput: false
  },
  ADDRESS: {
    key: "ADDRESS",
    label: "Search Address",
    needsInput: true
  },
  MAP: {
    key: "MAP",
    label: "Pick on Map",
    needsInput: false
  },
  FAVORITE: {
    key: "FAVORITE",
    label: "Use Favorite",
    needsInput: false
  }
};

That gives you a master source of truth.

Then build one master handler

Instead of separate half-duplicated functions like:
	•	setLocationFromGPS()
	•	setLocationFromAddress()
	•	setLocationFromMap()
	•	setLocationFromFavorite()

you can route them through one main function:

async function setLocation(method, payload = {}) {
  switch (method) {
    case "GPS":
      return await fetchLocationFromGPS();

    case "ADDRESS":
      return await fetchLocationFromAddress(payload.query);

    case "MAP":
      return await fetchLocationFromMap(payload.lat, payload.lon);

    case "FAVORITE":
      return await fetchLocationFromFavorite(payload.favoriteId);

    default:
      throw new Error(`Unknown location method: ${method}`);
  }
}

That is already cleaner.

Even better: use a strategy object

This is the cleaner, more modular version:

const LOCATION_HANDLERS = {
  GPS: {
    async fetch(payload) {
      return await fetchLocationFromGPS();
    }
  },
  ADDRESS: {
    async fetch(payload) {
      return await fetchLocationFromAddress(payload.query);
    }
  },
  MAP: {
    async fetch(payload) {
      return await fetchLocationFromMap(payload.lat, payload.lon);
    }
  },
  FAVORITE: {
    async fetch(payload) {
      return await fetchLocationFromFavorite(payload.favoriteId);
    }
  }
};

async function setLocation(method, payload = {}) {
  const handler = LOCATION_HANDLERS[method];
  if (!handler) throw new Error(`Unsupported location method: ${method}`);

  const location = await handler.fetch(payload);
  return applyLocation(location, { method, payload });
}

That way:
	•	UI buttons all call the same entry point
	•	all roads lead to one system
	•	easier to log/debug
	•	easy to add new methods later like:
	•	AIRPORT
	•	STORM_CELL
	•	CURRENT_MAP_CENTER

Best overall structure

I’d organize it like this:

const LOCATION_METHODS = {
  GPS: "GPS",
  ADDRESS: "ADDRESS",
  MAP: "MAP",
  FAVORITE: "FAVORITE"
};

const locationState = {
  current: null,
  saved: [],
  lastMethod: null
};

Then:

async function fetchLocationByMethod(method, payload = {}) { ... }

function applyLocation(location, meta = {}) { ... }

function saveLocation(location, meta = {}) { ... }

function loadSavedLocation(id) { ... }

So the flow becomes:
	1.	button clicked
	2.	setLocation(method, payload)
	3.	fetch/resolve location
	4.	apply it to app state
	5.	optionally save it

That is much more organized than having GPS/map/search/favorites all doing their own side quests in different corners of the file 🧭

Even more useful than arrays

You could store the method categories like this if you want:

const LOCATION_SOURCES = ["GPS", "ADDRESS", "MAP", "FAVORITE"];

But use that only for:
	•	dropdowns
	•	validation
	•	iterating available options

For real app logic, use objects with handlers.

My recommendation

Yes, do it. But make it a method registry + master location pipeline, not just two plain arrays.

A solid version would be:
	•	LOCATION_METHODS for constants
	•	LOCATION_HANDLERS for method-specific fetch logic
	•	setLocation(method, payload) as the master entry point
	•	saveLocation() as the master save entry point
	•	locationState.current and locationState.saved for state

That would make StormTracker feel way less spaghetti and way more radar-console-clean.
